1 Project Overview

Executive Summary

QuadruJuggle is a sim-to-real research project that teaches a Unitree Go1 quadruped to juggle a 40 mm ping-pong ball on a back-mounted paddle using reinforcement learning. The task is substantially harder than ball balancing: the robot must actively impart controlled upward impulses at each contact to sustain repeated bounces, requiring tight coordination between whole-body posture control and real-time ball trajectory estimation.

We use a hierarchical two-policy architecture: a low-level torso tracker (pi2, trained first) that maps 8D body-pose commands to joint targets, and a high-level ball planner (pi1) that reads ball state from a D435i depth camera + EKF and outputs the body-pose commands. Training runs in Isaac Lab (12 288 parallel envs) with RSL-RL / PPO. A noise-aware curriculum bridges sim-to-real by injecting realistic sensor noise during pi1 training, matching the noise statistics of our onboard EKF pipeline.

Current state (April 2026): pi2 has completed 8-stage curriculum (Stage H) and is frozen. pi1 is advancing through the juggling curriculum — sustained apex ≈ 9–11 cm through Stage B — with a critical algorithmic discovery (ball_release_velocity_reward) that unlocked the transition from passive balancing to active juggling. The EKF perception pipeline has reached production quality with 239/239 unit tests passing, chi-squared NIS gating, and phase-separated noise tuning for flight vs. contact phases.

QuadruJuggle trains a Unitree Go1 quadruped to bounce a 40mm ping-pong ball repeatedly on a 170mm disc paddle mounted on its back — a task we call quadrupedal ball juggling. 2 System Architecture

The system uses a hierarchical policy decomposition — pi2 (low-level torso tracker) is trained first and then frozen; pi1 (high-level ball planner) is trained on top using pi2 as a differentiable actuator. This separation means perception noise only affects pi1; pi2 is completely isolated from ball observations.

π₂ — Torso Tracker

Low-level controller. Maps 8D commands (height, tilt, angular rates, velocity) + 33D proprioception to 12 joint position targets at 50 Hz. Trained independently with domain randomization for sim-to-real. Frozen during pi1 training.

π₁ — Ball Planner

High-level controller. Observes ball position/velocity + proprioception, outputs 8D torso command. Trained with pi2 as a differentiable low-level actuator. Perception noise only affects pi1 — the hierarchy isolates pi2 entirely.

8D Command Interface

Normalised interface: h (trunk height 0.20–0.50m), ḣ (vertical rate ±1.5 m/s), roll/pitch (±0.5 rad), ω_roll/pitch (±4 rad/s), vx/vy (±0.5 m/s). Normalised to [−1, +1] before pi2 input.

Simulation Stack

Isaac Lab (RL environment framework) + Isaac Sim + PhysX 5 GPU physics. 12,288 parallel environments on a single GPU. Policy runs at 50 Hz (20 ms steps); physics at 200 Hz (5 ms). CCD enabled for ball tunnelling prevention.

3 Training Infrastructure

Algorithm: Proximal Policy Optimization (PPO)

Both pi1 and pi2 are trained with PPO via RSL-RL (Rudin et al., CoRL 2022). PPO's clipped surrogate objective prevents catastrophic policy updates — critical for a task where a single bad gradient step can permanently destabilize the learned gait: 

ratio          = π_new(a|s) / π_old(a|s)
surrogate      = −A · ratio
surrogate_clip = −A · clamp(ratio, 0.8, 1.2)      # clip_param = 0.2
loss_actor     = max(surrogate, surrogate_clip).mean()

Key lesson: entropy coefficient

entropy_coef = 0.01 caused noise_std to grow 0.1 → 1.64 over 400 iterations, then catastrophic policy collapse. The entropy term dominated the loss at high exploration noise. Fix: entropy_coef = 0.001 (10× reduction).

Key finding: ball_release_velocity unlocks sustained juggling

Without a release-velocity reward, pi1 learned a passive "cradle" strategy: keeping the ball stationary on the paddle for an extended episode rather than juggling. Adding ball_release_velocity_reward (weight +3.0) — rewarding upward ball speed at paddle contact — broke the local optimum. Apex height jumped from ≈2 cm (cradle) to 9.7 cm (active juggling) and sustained stably through 1,500 training iterations (policy iter_014). This reward is tapered and removed by Stage D to prevent over-aggressive throwing.

Key lesson: crouching local maximum

Using alive_upright(min_height=0.33) (gate returns 0 at boundary) allowed the robot to earn full 1500-step episodes crouching at exactly 0.33m with the ball sitting still — gradient vacuum. Fix: is_alive() with no gate; base_height penalty at 0.36m provides continuous upward gradient everywhere.

5 Perception Pipeline

The perception system follows the ETH noise-injection approach (Ma et al., Science Robotics 2025): rather than rendering cameras for 12,288 simulation environments (10–50× training slowdown), we inject a structured noise model onto ground-truth ball position, then filter with an EKF — the same code runs in both simulation and on real hardware.

EKF: 6D State Kalman Filter

The BallEKF maintains a 6D state [x, y, z, vx, vy, vz] per environment. Prediction uses ballistic dynamics with quadratic aerodynamic drag: 

ẍ = −c/m · ‖v‖ · vx       (drag, Cd=0.4, r=20mm, m=2.7g → c=0.112)
ÿ = −c/m · ‖v‖ · vy
z̈ = −g − c/m · ‖v‖ · vz   (gravity + drag)

Contact detection uses per-env contact flags to switch between two process noise covariance matrices: low-Q during free flight, high-Q (50× larger) during contact, when ball momentum changes abruptly.

NIS Gating (Chi-Squared Outlier Rejection)

Each measurement's Normalized Innovation Squared (NIS) is computed before the EKF update step. Measurements exceeding the 99th percentile of a χ² distribution with 3 DOF (threshold = 11.345) are rejected — preventing detector glitches or multi-ball confusion from corrupting the state estimate: 

NIS = νᵀ · S⁻¹ · ν        (ν = measurement innovation, S = innovation covariance)
gate = NIS < 11.345         (chi-squared 3DOF 99th percentile)
# first 50 updates per env skipped (warm-up: velocity P hasn't converged yet)

7 Sim-to-Real Plan

The sim-to-real gap has five components, each addressed by a dedicated phase:

  • A ✓
    Privileged training (pi2 + pi1)

    Train and validate both policies with ground-truth ball state. Pi2 complete; pi1 juggling Stage B, advancing.

  • B
    Domain Randomization + Latency Modeling

    Mass ±15%, friction [0.4,1.5], motor gains ±20%, joint friction ×[0.5,1.5], pushes ±0.3 m/s, external wrench ±2N/±1Nm. Action delay 5ms, obs delay 20ms. Retrain pi2, then pi1.

  • C ✓
    EKF Perception-Aware Training

    Replace GT ball obs with EKF estimates from noise-injected measurements. Full pipeline implemented (239 tests). Ready for pi1 retraining in noise-aware mode.

  • D
    Hardware Preparation

    Order D435i camera. Design + 3D-print 45° upward-facing bracket. Mount 170mm paddle at 70mm above trunk. Calibrate camera intrinsics + paddle rigid transform.

  • E
    System Identification + Calibration

    Actuator sine-sweep to measure bandwidth (critical: <10Hz would prevent 50Hz juggling). Ball restitution drop test. Real camera noise model calibration (≥500 measurements).

  • F
    Progressive Deployment

    Level 0: comms check. Level 1: pi2 manual commands. Level 2: circle walking. Level 3: pi2+pi1 with external ball tracking. Level 4: onboard perception. Level 5: full juggling.

8 Lessons Learned

Six reward-shaping, PPO, and simulation bugs were discovered and fixed during development. Each created a degenerate local optimum that required careful diagnosis to escape.

🦿 Clamped height gate → crouching local max
Bug

alive_upright(min_height=0.33) returns 0 at exactly the threshold. All height-gated rewards also return 0 → gradient vacuum → robot crouches at 0.33m and earns full 1500-step episodes with the ball sitting motionless.

# BAD — zero gradient at boundary
alive = height > 0.33   # binary step
Fix

Use is_alive() (unconditional) + base_height penalty with min_height=0.36m, which provides a continuous upward gradient from any height.

# GOOD — continuous gradient
base_height_penalty = relu(0.36 - trunk_z)  # ✓
Why it matters

A clamped gate produces zero gradient both above and below the threshold. Any near-optimal policy that discovers the gate never receives signal to improve further.

💥 entropy_coef=0.01 → entropy explosion
Bug

With entropy_coef=0.01, the noise standard deviation grew from 0.1 → 1.64 over 400 iterations, then policy catastrophically collapsed in an overnight run (iter_049 data).

# BAD — too strong entropy incentive
entropy_coef = 0.01   # default RSL-RL value
Fix

Reduce to entropy_coef=0.001 (10× reduction). The entropy term no longer dominates the total PPO loss at high noise scales.

# GOOD
entropy_coef = 0.001  # prevents runaway noise
Why it matters

PPO maximizes entropy as a regularizer; the default coefficient is calibrated for locomotion tasks with small action spaces. Manipulation tasks with larger observation variance require a lower coefficient to stay within the stable regime.

📉 Gaussian joint tracking std=0.5 → flat tail at crouch
Bug

exp(-Σ(q - q₀)² / 0.5) saturates near 0 when joints deviate 0.4 rad from default. No gradient to pull the policy back from crouched configurations.

# BAD — gradient ≈ 0 at 0.4 rad deviation
joint_tracking = exp(-sq_sum / 0.5)
Fix

Widen to std=2.0 → exp(-sq_sum / 2.0) ≈ 0.77 at 0.4 rad deviation, providing strong gradient everywhere including deep crouch.

# GOOD — ~0.77 at 0.4 rad off default
joint_tracking = exp(-sq_sum / 2.0)  # std=2.0
Why it matters

Gaussian rewards are attractive for their smooth gradients, but narrow σ creates a dead zone outside the mode. For posture recovery the reward must stay informative even at large deviations.

🤸 Joint randomisation at spawn → airborne at reset
Bug

reset_joints_by_scale(range=(0.85, 1.15)) occasionally spawns the robot with legs too short to reach the ground, causing immediate fall and wasted rollouts.

# BAD — legs may not reach ground
position_range = (0.85, 1.15)
Fix

Set position_range=(1.0, 1.0) (no randomisation) until standing is reliably learned. Reintroduce joint DR only after Stage C+ curriculum.

# GOOD — deterministic spawn stance
position_range = (1.0, 1.0)
Why it matters

Curriculum learning requires stable initial conditions. Adding joint noise too early means the policy must simultaneously learn to stand and to juggle — two skills that interfere.

🎲 init_noise_std=1.0 → large torques at spawn
Bug

With init_noise_std=1.0 and joint scale=0.25 rad, initial actions are ±0.25 rad — large torques that destabilise standing from the very first step.

# BAD — ±0.25 rad random torques at init
init_noise_std = 1.0   # default
Fix

Reduce to init_noise_std=0.1. The robot stays near the default standing pose at initialisation, giving the policy a stable starting point.

# GOOD — policy near standing at init
init_noise_std = 0.1
Why it matters

RSL-RL's default is tuned for locomotion where the robot is already moving. Balancing tasks start from a static pose; a large initial noise std immediately creates unrecoverable falls before any learning signal is received.

⏱ Trunk-height termination caps episode length
Bug

A grace_steps=50 trunk-height termination ended every episode exactly at step 50. The policy never experienced longer rollouts and could not learn long-horizon juggling behavior.

# BAD — all episodes end at step 50
terminate_if_height < 0.30m, grace=50
Fix

Remove trunk-height termination entirely. Use ball_below_paddle + continuous base_height penalty instead — this lets episodes run to 1500 steps while still penalising low posture.

# GOOD — base_height penalty, no hard kill
base_height_penalty = relu(0.36 - trunk_z)
# terminate only on ball_below or ball_off
Why it matters

Hard termination on height is a seductive safety net, but it creates a feedback loop: the robot crouches → terminates early → never learns that sustained upright posture is needed for ball control.

Method 1 — Direct Override (Recommended First Step)

Override pi2's cmd[6], cmd[7] after pi1 runs, before env.step(). Pi1 continues to control all other channels (height, roll/pitch, angular rates). Requires zero retraining; validated immediately with a backend="zero" smoke test (output must match baseline). 

Joystick (USB/gamepad)
    │  vx_user, vy_user  [m/s, clamped ±0.30]
    ▼
UserVelocityInput.get_cmd()
    │  vx_norm, vy_norm  [-1, +1]   (divide by 0.5)
    ▼
CommandMixer.mix(pi1_cmd, vel_user)
    │  replaces cmd[6], cmd[7] only; channels 0–5 pass through
    ▼
TorsoCommandAction.process_actions(mixed_cmd)  ← unchanged
    │  runs frozen pi2
    ▼
12D joint targets → robot

User safety cap: ±0.30 m/s (60% of pi2 training max). Velocity ramps on/off over 1 s to prevent jerk at spawn. Implemented in play_teleop.py (5-line addition to existing play loop).

Method 2 — Residual Velocity (Recommended Production)

Pi1 is fine-tuned to accept [vx_user, vy_user] as two additional observations (40→42D). Pi1 outputs residual velocity corrections; the final pi2 command is vx_final = clamp(pi1_out[6] + vx_user_norm, −1, +1). Pi1 learns to compensate for ball drift relative to the user's base motion.

Based on HiLMa-Res (Shi et al., IROS 2024): bounding pi1's output to a residual prevents the manipulation planner from overriding locomotion stability — user intent is guaranteed to be respected to first order. Training: zero-pad the new obs channels, hot-start from Stage G checkpoint at LR=1e-4 for 200 iters, then standard curriculum for 600 more.

Expected improvement over Method 1: at 0.20 m/s user velocity, pi1 adds ±0.10 m/s correction for ball drift → ~50% fewer ball-loss events vs. Method 1.

Method 3 — Multi-Mode Conditioning (Optional)

Pi1 obs extended by 3D: [mode_bit, vx_target_norm, vy_target_norm]. Mode 0 = balance (current); Mode 1 = locomote (user velocity active). A mode-conditional reward scales the apex reward to 30% in locomote mode while adding a velocity-tracking reward. Based on HOVER (Zhuang et al., CoRL 2024) unified mode-conditioned whole-body controller. Recommended only if Method 2's graceful degradation at ≥0.30 m/s is insufficient for the target demo scenario.

Integration note (Perception): Method 1 is perception-independent. Methods 2 & 3 should hot-start from a checkpoint that has already stabilised under full-noise EKF observations. Training with fresh noise AND fresh velocity curriculum simultaneously risks double-destabilisation.

Stage D Plateau

Both oracle and D435i policies converge to a "safe balance, don't bounce" equilibrium at Stage D (target=0.20m, σ=0.08m). The breakthrough came from ball_release_velocity_reward (+3.0), which provides a positive gradient for throwing the ball, preventing collapse to passive balance. With this reward, the policy sustained juggling through curriculum transition (Stage A → B) with apex_reward ≈ 9.7–10.5 for 1 000+ iterations.

Rationale: Stages A–C use clean oracle observations so the policy can discover throwing without fighting noisy inputs. Stages D–F ramp noise gradually (25 / 50 / 75% of full D435i σ) to avoid catastrophic forgetting at the noise-introduction boundary. Stages G+ switch to the full EKF pipeline, which provides filtered estimates with one-step observation latency (realistic for a 30 Hz global-shutter camera).

NIS–curriculum link: The EKF NIS validation (perception iter_049) showed NIS ≈ 0.44 — the filter is over-conservative (R too large). This means the noisy ball observations fed to pi1 are smoother than the noise model predicts, which is beneficial for robustness: the policy trains against a pessimistic noise model and deploys with a slightly better real sensor.
